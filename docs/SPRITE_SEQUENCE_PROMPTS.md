# 序列帧生成 Prompt 清单

本文件基于当前项目里已经存在的 `assets/sprites/units` 目录和 `data/units.json` 整理，目标不是再写一份零散 prompt，而是把“现有角色资源”“逻辑 id”“视觉目录”“动作类型”统一起来，方便后续批量补图、补动作、补技能。

适用范围：

- 我方单位序列帧
- 敌人序列帧
- Boss 序列帧
- 建筑序列帧

## 统一硬规则

统一要求：透明背景 PNG；普通角色、敌人、建筑每帧 `128x128`，Boss 每帧 `192x192`；三分之二俯视伪 3D 视角；暗色战术奇幻科幻风格；主体居中偏下；底部中心作为锚点；缩小到 `64px` 地图格后轮廓仍清楚；角色完整身体可见，脚部尽量可见；光照方向统一；特效克制，不遮挡主体。

额外要求，需贴近项目里当前已经落地的小人风格：

- 比例是“紧凑战棋精灵”，不是写实 6-7 头身，也不是幼态卖萌 Q 版。
- 允许略夸张的头肩、武器、盾牌、法器轮廓，用来保证 `64px` 下的识别度。
- 角色外轮廓必须比内部花纹更重要，优先读出职业和武器。
- 玩家单位边缘更整洁，敌人边缘更污染、更破损、更危险。

统一不要：

- 文字、数字、Logo、水印、UI 边框、背景、地面、透视背景板
- 纯黑剪影、正面头像、半身裁切、写实 3D 渲染、照片质感
- 过曝光效、全屏特效、巨大弹幕、看不清本体的能量团
- 多角色拼图、角色变形、每帧重新设计服装或武器

## 目录与命名规则

推荐统一采用下列正式结构：

```text
assets/sprites/units/<visual_key>/<action>/<frame_prefix>_<action>_000.png
```

其中：

- `<visual_key>`：目录名，和 `units.json` 里的 `visual_key` 一致
- `<action>`：动作目录名，单位建议使用 `idle`、`attack`、`skill`、`deploy`、`death`
- `<frame_prefix>`：建议与 `visual_key` 一致；如果为了兼容逻辑 id 需要使用 id 前缀，也尽量统一到底

建议后续新资源统一写法：

```text
assets/sprites/units/skadi/idle/skadi_idle_000.png
assets/sprites/units/exusiai/attack/exusiai_attack_000.png
assets/sprites/units/logos/skill/logos_skill_000.png
```

说明：

- 当前项目里少量旧文件存在 `id` 与 `visual_key` 混用情况，例如 `guard_t1` 对应 `skadi`。
- 新增资源时建议统一采用 `visual_key` 作为目录名和文件名前缀，减少后续导入混乱。
- 如果先生成的是 `cast`，落地时可以映射或复制到 `skill`；项目调试预览目前更偏向使用 `skill` 目录名。

## Prompt 书写母版

下面几段建议作为所有单位 prompt 的公共前缀或拼装段，先锁风格，再填角色词。

### 公共风格前缀

```text
暗色战术奇幻科幻塔防游戏，手绘 2D 精灵，三分之二俯视伪 3D 视角，紧凑战棋小人比例，
强调职业轮廓和武器识别度，深蓝灰阴影，低饱和金属和布料，少量职业色点缀，
透明背景，主体居中偏下，底部中心锚点，缩小到 64px 地图格后轮廓仍清楚。
```

### 公共负面约束

```text
不要文字、数字、Logo、水印、背景、地面、UI 边框、裁切身体、过强光效、纯黑剪影、
可爱幼态 Q 版、正面头像、写实 3D、照片质感、每帧服装武器不一致、超出画布的特效。
```

### 动作统一说明

- `idle`：只做呼吸、衣摆、装备、法环、灯光等小幅运动，脚点稳定
- `attack`：包含起手、出手、命中感、收招，动作要读得清，不要做太花的镜头感
- `skill`：更强调角色招牌能力，特效比 `attack` 明显，但不能盖住主体
- `deploy`：角色自上而下或快步入场后定住，适合 4-6 帧
- `death`：倒下、消散、熄灭、撤离皆可，避免血腥

## 使用建议

建议后续工作顺序：

1. 先按本文件里的“职业专属 prompt”生成统一角色参考图。
2. 选定每个角色最稳定的一张作为参考，再生成 `idle`、`attack`、`skill`。
3. 导入项目前先统一画布大小、锚点、文件名前缀。
4. 进入游戏或调试预览检查帧间抖动、脚底漂移、武器丢失、特效遮挡。
5. 优先补齐基础职业，再做高阶单位的技能动作。
