# 动态出怪口与波次模板组化 设计稿

> 状态：已批准；v1 已实现（2026-06-11，分支 fix/map-popup-floating-layer，含按波分段预览、迷雾常显口标记、夜间词缀横幅）。v2（5 候选口）未实现。本文取代《肉鸽构筑与战斗优化方案》§4.5 与 §8.1 P2-4 的原"动态刷怪点"方案。
> 实现前先读根目录 `AGENTS.md`；数据结构变更需同步 `docs/DATA_SCHEMA.md`。

## 1. 问题

1. 怪物行进方向开局即定，玩家只需固定防守路线预览给出的几条路，夜与夜之间缺乏空间维度的新决策。
2. 波次模板把出怪口写死为 S1/S2/S3（`data/wave_templates.json` 的 `entries[].spawn_key`），任何增减出怪口的机制都会让既有模板数据失效。

## 2. 设计总览

把"出什么怪"（模板层）与"从哪出"（出怪口层）解耦：

- 模板内部改为若干**组**（group）：同种怪的一簇，保留 `time/count/interval` 节奏字段；
- 组不再指定出怪口，改带**进攻角色 `lane`**（`main` / `flank` / `any`）；
- 每天清晨结算：确定当晚**活跃口集合**与**每波主攻口**，把各组按角色发牌到具体口，随敌情预览一并公示；夜间严格按结算结果执行，不重摇。

玩家侧不新增任何机制：感知面仍然是"清晨读预览 → 部署"。所有复杂度收在数据层与解析器里，以"每晚不一样"的形式被感知，而不是以"要学一个新系统"的形式。

分组的目的（用户明确要求）：防止随机分布过于均匀。随机的只是**方向**，进攻的**形状**仍由模板作者控制——main 组保证有主攻方向，flank 组保证有侧翼冷箭，压力分布"有形态"（lumpy）而非糊开。

## 3. 数据结构

### 3.1 模板 schema 变更

`entries` → `groups`，`spawn_key` → `lane`。以现有 `arts_eclipse` 为例：

```json
{
  "id": "arts_eclipse",
  "tier": "mid",
  "key_enemies": ["caster"],
  "groups": [
    { "time": 0.0,  "enemy_id": "soldier",         "count": 5, "interval": 0.75, "lane": "main" },
    { "time": 4.0,  "enemy_id": "caster",          "count": 3, "interval": 1.4,  "lane": "flank" },
    { "time": 8.0,  "enemy_id": "armored_soldier", "count": 4, "interval": 1.0,  "lane": "main" },
    { "time": 14.0, "enemy_id": "shieldguard",     "count": 1, "interval": 0.0,  "lane": "any" }
  ]
}
```

`lane` 语义：

| 取值 | 落口规则 |
|---|---|
| `main` | 该波主攻口 |
| `flank` | 从该波**非主攻**的活跃口中独立 seeded 抽一个；活跃口仅 1 个时回退 main |
| `any` | 全体活跃口加权随机（v1 等权） |

作者约定（数据规范级，不做运行时硬断言）：

- 每模板至少 1 个 `main` 组，main 组合计强度占当波多数——保证"主攻方向"名副其实；
- Boss 所在组固定 `lane: "main"`，Boss 永远从主攻方向登场；`key_enemies` 不强制（预览的 key 提示与 lane 无关，骚扰型 key 如远程/飞行常作 flank）；
- 飞行/远程骚扰组优先 `flank`，杂兵填 `any`。

迁移：现有 15 个模板的 entries 已天然按"同种怪一簇"组织，逐条把 S1/S2/S3 按上述约定映射为角色即可（机械替换 + 一次作者判断）。

### 3.2 出怪口层

- 地图生成放 **5 个候选口**（seeded，满足互相最小间距 + 到核心连通校验），全部作为**可见地形**从第 1 天暴露给玩家，支持长线的墙/塔规划，避免"凭空开口"的突袭感；
- 每晚活跃口**数量**由配置表驱动（与 `WAVE_TIERS_BY_DAY` 同款风格，调节奏只改这张表）：

```gdscript
const ACTIVE_GATES_BY_DAY := { 1: 2, 3: 3, 5: 4 }   # 占位值
```

- 活跃口**集合**每晚 seeded 抽取，清晨随预览公示；非活跃口当晚沉默（预览中灰显）；
- 原 P2-4"第 4 天预告、第 5 天新增一口"由该表自然表达（day 5 活跃数 +1），不再是独立机制。

### 3.3 可见性实现：标记穿透迷雾（2026-06-10 用户评审补充）

现状：出怪点标记可见性绑定格子 `discovered` 状态（`map_manager.gd:394`），开局完全隐藏。"候选口第 1 天起可见"必须用**标记穿透迷雾**实现，**禁止**把口的格子直接置为已探索：

- 仅解绑出怪口标记（`spawn_point_view`）与 `is_discovered` 的可见性绑定，使其常显；格子本身保持 `discovered = false`；
- 探索约束不受影响：探索仍只能从已探索区四向连续扩展（`day_manager.try_explore` 的 `has_discovered_neighbor` 校验），玩家**不能**从出怪口开始探索；
- 随机事件"探索前沿"落点算法不受影响：`random_event_manager._pick_event_spawn_cell` 的 frontier/fog/discovered 分类完全基于 `discovered`，口的标记可见与否与之无关；
- 迷雾中的敌人照旧隐身（`enemy_actor._refresh_fog_visibility`）；路线投影照旧只绘制已探索路段（`map_root_view.gd:760`）——**口的位置已知，但其进军路线穿过迷雾时仍是未知情报**，探索保留侦察价值。

该实现自 v1 即对现有 3 个口生效（"主攻口=北门"的公示需要玩家在地图上找得到北门）。

## 4. 解析与公示流程

清晨进入白天阶段时（与现有夜计划解析同一时机）：

1. 由 `ACTIVE_GATES_BY_DAY` 与 seed 确定活跃口集合；
2. 逐波抽**主攻口**——沿用 `night_template_resolver.gd` 的 `run_seed|day|wave` 哈希习惯；**主攻口每波重抽**（已拍板），一晚 2-3 波时夜内形成方向轮转，给再部署与突袭盟约真正的舞台；
3. 按 `lane` 把各组发牌到具体口（flank/any 的独立抽取在 seed 中加入组序）；
4. 解析结果（含每组最终落口）写入夜计划；预览按"波 × 口 → 组构成"展示，路线投影覆盖所有活跃口；
5. 夜间执行严格按解析结果刷怪——**预览即契约，夜里绝不重摇**。

## 5. 兼容性

- **spawn_surge（西门失守）词缀**：语义改为修改主攻口份额/权重，比现行"按口乘数"实现更顺；其余词缀不受影响。
- **路径校验**：`path_service` 已有 spawn→core 连通性与 `no_path`/`core_enclosed` 状态，活跃集抽取与发牌跳过非法口。
- **enemy_choices（P3-3）**：组内字段，原样兼容，后续在组粒度上做加权替换。
- **强度系数（方案 §4.4）**：作用于组展开，与口分配正交。

## 6. 落地切分与验收

### v1：模板组化 + 角色分配（口数固定 3，不动地图）

范围（含同步 UI，2026-06-10 用户评审补充）：

1. 数据迁移：15 个模板 entries → groups、spawn_key → lane；
2. 解析器：活跃口集合 + 每波主攻口 + 按 lane 发牌（清晨结算，决定性 seed）；
3. 刷怪执行消费解析结果；
4. **预览按"波 × 口"分组展示**：`wave_manager.get_night_preview()` 的 `wave_summaries` 已带 `wave_index` 但 UI 未消费（`combat_hud.gd:528` 仅聚合展示）——本项补齐，敌情卡片按波分段并标注落口与主攻口；
5. **出怪口标记穿透迷雾常显**（见 §3.3）；
6. **夜间战斗 HUD 显示当晚词缀清单**（含事件临时追加项）：现状词缀仅白天预览可见（`combat_hud.gd:979-987`），入夜后无任何展示——按用户要求"夜晚临时 tag 玩家必须显式看到"并入本项。

触点：`wave_templates.json`、`night_template_resolver.gd`、`wave_manager.gd`、`combat_hud.gd` / `combat_hud_controller.gd`（预览分波展示 + 夜间词缀）、`map_manager.gd`（标记可见性解绑）、`docs/DATA_SCHEMA.md`。

验收（headless）：

- 同 seed 下整夜解析（含每波主攻口、每组落口）决定性；
- ≥2 活跃口时 flank 组落口 ≠ 该波主攻口；
- 预览与实际刷怪一致（既有断言扩展到口维度）；
- 探索约束与事件前沿落点不受标记常显影响（断言出怪口格子仍 `discovered == false`）；
- 既有四套件（night_template_flow / night_waves_affixes / relic_draw / contract_events）通过。

验收（人工/视觉）：

- 开局地图上能看到 3 个出怪口标记（迷雾中）；
- 白天预览能区分"第 2 波主攻北门"级别的信息；
- 夜间能看到当晚全部词缀（含战争赌局/军械库临时追加的）。

### v2：候选口 5 + 活跃集配置表

触点：`map_generator.gd` / `map_manager.gd`（候选口生成与注册）、`ACTIVE_GATES_BY_DAY`、预览 UI（活跃/沉默口状态）。

验收（headless）：

- 5 候选口生成满足间距与连通约束；
- day1 活跃 2 口、day3 起 3 口、day5 起 4 口；活跃集公示与实际一致；
- 非活跃口整晚无刷怪。

## 7. 范围外：P0/P1 UI 补课清单（2026-06-10 审计）

本设计评审时顺带审计了已落地 P0/P1 功能的前端可见性，以下缺口**不在本特性范围内**，已记入《肉鸽构筑与战斗优化方案》§8.1 的 UI 补课清单：祭坛灌注盟约 tag 的干员详情/卡片展示、遗物三选一槽位来源标注、事件 requires 不足的原因提示、波间倒计时、活跃事件数量指示。与本特性直接相关的两项（预览按波分段、夜间词缀显示）已并入 v1 范围。

## 8. 已否决与显式延期

- **否决：威胁源/裂隙系统**（迷雾中种子生长为新口 + 白天远征封堵/采集）。否决理由：为玩家侧引入全新实体与动词，超出当前复杂度预算。其"探索情报价值"目标由可见候选口部分覆盖。
- **显式延期**（本轮迭代完成后，先审视地形机制与地图生成机制，再回头设计）：
  - 口开关的更多驱动源：随机事件（如坍塌隧道封口）、玩家主动操作、夜数推进之外的触发；
  - 活跃集的连续性/反连续性权重（是否偏向延续昨晚主方向）——v2 玩测后定；
  - 出怪口与地形（隘口、高地等）的耦合设计。
