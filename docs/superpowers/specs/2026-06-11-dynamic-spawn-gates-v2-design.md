# 动态出怪口 v2：五候选口 + 活跃集 + 开关动态化 设计稿

> 状态：已批准（2026-06-11 用户拍板"方案 A"，玩家封口列为必做）。
> 前置：v1（模板组化 + lane + 每波主攻口，PR #227）已合入 dev。本稿基于 v1 的"公示即契约"架构延伸。
> 数值全部为占位符（项目未做平衡盘），实现时取整即可。

## 1. 目标

1. 候选口从 3 提到 5，方向分散由构造保证（不再靠运气）；
2. 活跃口集合随天数推进**单调扩张**（战线扩张叙事），用户既定验收：day1 两口 → day3 三口 → day5 四口；
3. 口的开关接入三类来源：**夜晚数量推进**（日程表）、**随机事件**（封口契约 / 开口赌约 / 异动警报）、**玩家主动操作**（付费封口一晚，必做）；
4. 红线：**口集合白天可变、夜内冻结**——黄昏预览的活跃集 == 夜间实际刷怪的活跃集（公示即契约延伸）。

## 2. 机制规格

### 2.1 候选口等弧放置（map_generator）

替换 `_place_spawns` 的"边缘洗牌 + 贪心"：把周长（30×30 = 116 边缘格）按顺时针展开成一维序列，均分为 `spawn_count` 段，每段只在**中间 60%** 范围内 seeded 抽一格。效果：任意两口沿周长至少隔约 9 格、方向分散构造保证、无凑不满失败分支、对任意口数通用。口格规则不变（强制平原、不可建、保持未探索、apron 保护）。删除已失效的"离核心曼哈顿 ≥12"约束（边缘格距中心核心最近即 15）。`map_generation.json`：`spawn_count` 3→5。距四角 ≥3 格（避免角落口贴双边）。

### 2.2 活跃集解析（night_template_resolver 纯静态，无存档状态）

- **激活序**：`activation_order = seeded_shuffle(sorted(all_gates), hash("gate_order|%d" % run_seed))`——每局一个固定顺序；
- **日程表**（数据驱动，占位值）：`ACTIVE_COUNT_BY_DAY = {1: 2, 3: 3, 5: 4, 7: 5}`（阶梯取 ≤day 的最大键）；
- **当日基础活跃集** = 激活序前 `count(day)` 个；单调、决定性、任意一天可从 seed+day 纯函数重算；
- **有效活跃集** = `(基础 ∪ extra_open) − closed`，下限 1（清空时保留激活序第一位）；
- 新增静态函数：`resolve_activation_order(gates, run_seed)`、`resolve_active_gates(gates, run_seed, day, closed, extra_open)`。lane/主攻口解析（v1）的 `active_gates` 入参改喂有效活跃集，其余不动。

### 2.3 一夜覆盖项（RunState）

- `night_gate_closed_keys: Array[String]`、`night_gate_extra_open_keys: Array[String]`，**enter_day 清空**；
- 只做一夜有效：永久变更会破坏无状态激活序，且一夜语义让事件可重复出；
- 所有变更只发生在白天阶段（day_manager / 事件系统已有阶段门控，结构性满足夜内冻结）。

### 2.4 玩家主动封口（必做）

- 白天点击**活跃口**标记 → 弹窗"封堵一晚"：花费 **4 石 + 6 AP**（占位），每天最多 1 次，封后有效活跃集不得 < 1（UI 置灰 + RunState 校验双保险）；
- 走请求-结果事件对（仿 P2-5 升星：`request_seal_spawn_gate` / `seal_spawn_gate_result`），RunState 整体预校验（白天 / 资源 / AP / 次数 / 下限）；
- **不要求探索到口**（远口探索链 20+AP 会让封口不可用）；语义=总刷怪量不变、怪改派他口——是导流不是减伤。

### 2.5 事件接入（events.json，2-3 个）

1. **塌方契约**（封口向）：付资源指定一个活跃口今晚沉默（写 closed）；
2. **开口赌约**（风险换收益）：自愿让激活序下一个沉默口今晚提前激活（写 extra_open），换资源/额外三选一；
3. **异动警报**（威胁向，可选做）：随机事件宣告某沉默口今晚提前激活，黎明公告。
   requires 校验沿用契约系统；选项动态生成可参考祭坛（按当前口集合出选项）。

### 2.6 消费端（wave_manager）

`_active_spawn_keys()` 从"map_manager.get_spawn_keys() 全集"改为"resolve_active_gates(...) 有效活跃集"；预览与运行时共用同一函数（单点真源），v1 的 lane 发牌、主攻口重抽、spawn_surge 词缀自动适配。沉默口整晚零刷怪。

## 3. UI 同步（与机制同 PR，不留欠账）

1. 地图标记两态：活跃（亮）/ 沉默（灰暗），当晚有变化的口挂角标（新开 / 已封，可用代码绘制或 icon_text，不依赖新美术）；
2. 黎明 toast / 晨报行：「战线扩张：S4 今晚加入进攻」「S2 已封堵一晚」；
3. 夜晚预览面板头部加活跃集行：「今晚活跃口：S1 S3 S4（S2 已封堵）」；分波卡片与主攻标注沿用 v1；
4. 口标记点击弹窗：状态、封口按钮（费用、不可封原因置灰提示）；
5. 路线预览维持迷雾裁剪；沉默口不画路线；
6. 词缀横幅不掺口信息（避免过载）。

## 4. 测试验收（headless）

- 等弧放置：5 口在边缘、段内分散（多 seed 断言任意两口周长距 ≥ 阈值）、确定性、口格不变式（plain/不可建/未探索）沿用 v1 断言；
- 激活序：同 seed 决定性；day 推进单调（前 N 天活跃集是第 N+1 天的子集）；日程表阶梯正确；
- 覆盖项组合：closed/extra_open 与基础集的并差、下限 1 钳制、enter_day 清空；
- 封口动作：双资源 + AP 整体预校验、每日次数、阶段门控、失败原因码；
- 夜内冻结：白天封口后 `get_night_preview` 与夜间实际刷怪的口集合一致（契约断言扩展）；
- 事件：塌方/开口赌约 apply 后覆盖项生效、预览刷新；
- 既有套件（七个）全绿；v1 单口回退路径天然覆盖"活跃集=1"。

## 5. 范围外 / 接口说明

- 口开关的**地形耦合**（封没有炮位的口等）由地形包迭代消费，本稿只保证 sectors/口元数据可查询；
- 等弧放置算法将来被地形包骨架生成器（S2）原样吸收，接口即 `spawn_count` + 周长分段参数；
- spawn_surge 词缀"主攻口份额"语义重做仍是既有 follow-up，不在本稿；
- 跨天 gate 种子隐式依赖 RunState.day 的现状保留（v1 注释已标）。
